# 收尾流程

当用户表示阶段性工作完成（或你判断一个功能阶段结束），自动执行以下流程。

## 步骤

### 1. 检查 git 状态

```bash
git status --short
```

列出所有改动的文件。如果没有任何改动，告知用户并结束。

### 2. 检查测试覆盖

对每个改动的源文件（排除 locales、docs、config），检查是否有对应的 `.test.ts` 文件：

- `src/stores/xxx.ts` → `src/stores/xxx.test.ts`
- `src/components/xxx/yyy.vue` → 搜索 `**/yyy.test.ts`
- `src-tauri/src/xxx.rs` → `src-tauri/tests/xxx_test.rs` 或 `src-tauri/tests/integration_test.rs`

如果改动了核心逻辑但测试文件不存在或未更新，**主动补充测试**：
- vitest 单元测试（`describe` + `it`）
- 验证关键交互和行为

### 3. 检查 CLAUDE.md

对比本次改动的实际内容与 CLAUDE.md 中的描述：

- **设计决策**：新增了设计决策吗？→ 补充到「关键设计决策」章节
- **功能索引**：新增了功能或改变了现有功能的文件位置？→ 更新「功能索引」表
- **模块职责**：新增了模块或命令？→ 更新 Rust 模块职责表
- **数据流**：改变了数据流？→ 更新核心数据流图

如果 CLAUDE.md 不需要更新，明确说明原因。

### 4. 检查 docs/ 计划文档

```bash
ls docs/*.md
```

如果此前有相关的计划文档（如 `PHASE*-PLAN.md`），对照计划核对完成情况，标记已完成项。

如果没有对应的计划文档且改动较大，建议是否需要补写计划。

### 5. 分批提交

按功能分组，使用 Conventional Commits 格式，提交信息使用中文：

| 前缀 | 场景 |
|------|------|
| `feat(模块):` | 新功能 |
| `fix(模块):` | Bug 修复 |
| `refactor(模块):` | 重构 |
| `chore(模块):` | 构建/配置 |
| `docs:` | 文档 |

模块标签：`chat`, `files`, `session`, `settings`, `rust`, `mcp`, `ui`, `i18n`, `test`, `docs`

分批原则：
- Rust 后端改动单独一批
- 前端功能改动按功能分批
- i18n 和 CLAUDE.md 可随功能提交
- 测试文件随对应源文件一起提交

执行提交前，**必须向用户展示分组方案并确认**：

```
分批方案：
1. fix(rust): 描述 — 文件1, 文件2
2. feat(mcp): 描述 — 文件3, 文件4
3. feat(ui): 描述 — 文件5, 文件6

确认提交？
```

### 6. 生成总结

提交完成后，输出本次收尾的总结：

```
## 收尾完成

### 提交
- <hash> fix(rust): 描述
- <hash> feat(mcp): 描述

### 测试
- 新增/更新：xxx.test.ts
- 无需更新：yyy（纯样式改动）

### 文档
- CLAUDE.md：补充了 xxx
- docs/：无需更新
```

## 注意事项

- 不要在用户未确认的情况下提交
- 不要跳过测试检查——即使改动很小，也要验证
- 如果改动涉及多个不相关的功能，分批提交；如果紧密耦合，一个提交也可以
- 提交信息简洁明了，一句话说清楚改了什么
