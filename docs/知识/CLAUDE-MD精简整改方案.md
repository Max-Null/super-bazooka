# CLAUDE.md 精简整改方案

> 适用于任何 Claude Code 项目的 CLAUDE.md 优化。
> 目标：将项目 CLAUDE.md 从「全能参考手册」精简为「压缩后能存活的骨架」。

---

## 问题诊断

### CC 的上下文机制

```
会话开始
  ├── CLAUDE.md（项目级）── 全量加载，带 "may not be relevant" 免责
  ├── CLAUDE.md（全局级）── 同上
  ├── SessionStart Hook 注入 ── 无免责标签，最高优先级
  └── Memory 文件 ── 前 200 行加载，跨会话持久化

会话过长时
  └── /compact（自动压缩）
        ├── Hook 注入内容 ── ✅ 每次会话重新注入，不受影响
        ├── Memory ── ✅ 每次会话重新加载
        └── CLAUDE.md ── ❌ 被摘要压缩，关键规则可能丢失
```

### 臃肿 CLAUDE.md 的症状

| 症状 | 根因 |
|------|------|
| 压缩后模型忘记"禁止手搓轮子" | 这条规则和 300 行其他内容一起被摘要了 |
| 压缩后模型开始直接 `invoke()` | `tauri-bridge.ts` 封装规则被压缩掉了 |
| 压缩后 CSS 变量被硬编码替代 | UI 规范在 CLAUDE.md 里但压缩时丢失 |
| 新增文案缺 i18n | 中英双语的硬性要求被埋没在长文档里 |

### 判断标准：这条内容压缩后还需要吗？

| 内容类型 | 压缩后需要？ | 处理方式 |
|----------|-------------|----------|
| 项目是什么、入口文件在哪 | ✅ 必须 | 保留 |
| 最高优先级规则（禁止手搓轮子、i18n） | ✅ 必须 | 保留，放最显眼位置 |
| 构建/测试命令 | ✅ 需要 | 保留，精简 |
| 核心数据流（CC 怎么 spawn） | ✅ 需要 | 保留，精简为单段 |
| 设计决策（为什么这样做） | ⚠️ 核心 5 条需要 | 大部分移走 |
| UI 组件速查表（按钮 class、颜色变量） | ❌ 不需要 | 删除——grep 同类组件比查表可靠 |
| 功能索引表（哪个功能在哪个文件） | ❌ 不需要 | 删除——grep 比查表快 |
| 完整的目录树 | ❌ 不需要 | 精简到关键入口文件 |
| 语言规范细节 | ❌ 不需要 | 放全局 CLAUDE.md，项目级不重复 |

## 整改步骤

### 第 1 步：创建参考文档（搬出去）

创建 `docs/知识/设计决策参考.md`，把非关键内容搬过去：

- 完整的设计决策列表（CLAUDE.md 只保留 3-5 条核心的）
- 版本号管理流程
- 功能索引表（如果你真的需要它）

**原则：搬出去 ≠ 删除。需要时 Read 即可，不占 CLAUDE.md 配额。**

### 第 2 步：精简 CLAUDE.md（目标 ≤ 60 行）

只保留以下内容，其余全删：

```markdown
# 项目名 CLAUDE.md

> 一句话描述项目。

## 架构入口
关键文件路径（不列完整目录树，只列入口文件）
不确定时指向 docs/ 参考文档

## 核心数据流
最关键的一条数据流路径（用户操作 → 后端 → 返回）

## 🔴 最高优先级规则
3-5 条绝对不能违反的规则，每条 2-3 行
这是压缩后最需要存活的内容

## 构建与测试
精简的命令列表

## 关键设计决策
top 3-5 条，每条一行
完整列表指向 docs/知识/

## 阶段性完成必做
自动收尾流程（测试、文档、commit）
```

### 第 3 步：善用 SessionStart Hook（强化注入）

将行为规则（Ponytail、Karpathy、代码风格等）放到 `~/.claude/hooks/behavioral-rules.md`，通过 SessionStart Hook 注入。

**为什么这比写在 CLAUDE.md 里强：**
- Hook 注入没有 "may not be relevant" 免责标签
- 每次新会话都重新注入，不受 `/compact` 影响
- CLAUDE.md 可以专注于项目特定信息

Hook 配置示例（`~/.claude/settings.json`）：
```json
{
  "hooks": {
    "SessionStart": [
      {
        "type": "command",
        "command": "powershell -ExecutionPolicy Bypass -File \"C:\\Users\\<user>\\.claude\\hooks\\session-start.ps1\""
      }
    ]
  }
}
```

### 第 4 步：用 Memory 记录纠正与决策背景

以下内容适合放 Memory 而非 CLAUDE.md：

| 场景 | Memory 内容 |
|------|------------|
| 用户纠正过你 | "上次切换 Provider 时 base_url 被忽略，已修复为优先用户输入" |
| 选型决策 why | "选 SQLite 不选 JSON 文件是因为需要 WAL 并发写" |
| 已知坑位 | "Windows 下 Rust 测试需要特定 DLL，CI 环境需单独配置" |
| 命名规范反直觉 | "`claude_session_id` 是 CLI 返回的 ID，`ourId` 是前端 UUID" |

### 第 5 步：验证

压缩后能回答这三个问题就是成功的：

1. **能说出这是什么项目吗？** → 架构入口段
2. **能遵守最高优先级规则吗？** → 🔴 段
3. **能跑测试和构建吗？** → 构建与测试段

验证方法：
```bash
# 触发压缩
echo "/compact" | claude

# 然后问模型：
# "你现在还记得 cc-gui 有哪些硬性规则？"
# "你怎么跑测试？"
# 如果答不上来，说明骨架还不够精简
```

## 行数预算参考

| 项目规模 | CLAUDE.md 目标行数 | 说明 |
|----------|-------------------|------|
| 个人小项目 | 20-40 行 | 架构 + 规则 + 命令 |
| 中型项目 | 40-60 行 | 以上 + 核心设计决策 |
| 大型项目 | 60-80 行 | 以上 + 多模块入口 |
| 超过 100 行 | ⚠️ 需要精简 | 考虑拆到 docs/ 和 Hook |

## 常见反模式

| 反模式 | 为什么不好 |
|--------|-----------|
| 把 API 文档写在 CLAUDE.md 里 | 用 MCP 工具查官方文档，不要手写 |
| 把 CSS class 列表写在 CLAUDE.md 里 | grep 已有组件模板更准确 |
| 把完整的目录树列出来 | `ls` 或 Glob 比手写目录树更新更准 |
| 把每个设计决策都写进去 | 压缩时一条都活不下来，不如只留核心 5 条 |
| 三层格式、彩色标记、emoji 分组 | 压缩时格式丢失，标记白做。格式不如内容精简重要 |
| CLAUDE.md 和全局 CLAUDE.md 重复 | 全局放语言规范，项目级放项目特定规则 |
| **把参考文档搬出 CLAUDE.md 但不建立查找机制** | 压缩后模型不知道文件存在，也不知道什么时候该看它——见下方「关键问题」 |

## 关键问题：搬出 CLAUDE.md 的文档怎么被找到？

精简 CLAUDE.md 的本质是把内容"搬出去"（docs/、Memory、Hook）。但压缩后怎么办？模型不知道 `docs/知识/设计决策参考.md` 存在。

### 三层防线

建立冗余的查找触发机制，确保至少一层在压缩后存活：

```
压缩前                            压缩后
═══════════                       ═══════════
CLAUDE.md 触发规则                CLAUDE.md 指针（概率存活）
"改 .rs 前 Read 设计决策参考"

SessionStart Hook（不受影响）       SessionStart Hook（重新注入）
→ behavioral-rules.md              → behavioral-rules.md

Memory 前 200 行（不受影响）        Memory 前 200 行（重新加载）
→ "改核心模块前查阅设计决策"          → "改核心模块前查阅设计决策"
```

### 具体做法

**1. CLAUDE.md 加触发规则**

不是在 CLAUDE.md 里列文档路径，而是写"什么情况下必须去看什么文档"：

```markdown
### 动手前必做
1. Grep 已有实现
2. **修改 src-tauri/ 下 .rs 文件前** → Read docs/知识/设计决策参考.md
3. 写 UI 前 → Grep 同类组件模板
```

触发规则 > 文档索引。模型不需要记住所有文档，只需要记住"改 Rust 前该干什么"。

**2. Memory 写冗余指针**

每一份搬出 CLAUDE.md 的重要参考文档，在 Memory 里对应一条：

```markdown
---
name: design-decisions-reference
description: 修改 Rust 核心模块前必须查阅设计决策参考文档
type: reference
---

修改 src-tauri/ 下任何 .rs 文件前，必须先 Read docs/知识/设计决策参考.md。
Why: CLAUDE.md 精简到 86 行只保留 top 5 决策。压缩后如果触发规则丢失，这条 Memory 是最后防线。
```

Memory 每次会话重新加载，不受压缩影响。这是最稳的防线。

**3. 避免沉没**

以下情况需要两条 Memory 而不是一条：
- 改了核心模块 → 指向设计决策参考
- 改了 MCP 相关 → 指向 MCP 多源扫描规则（如果这条被移出 CLAUDE.md）

每条 Memory 只做一件事：一个触发条件 → 一个文档路径。
