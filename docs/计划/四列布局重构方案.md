# 四列布局重构方案

> 状态：待实施 | 2026-07-08

## 背景

当前页面 body 的四列布局是逐步加功能时"打补丁"堆出来的——先有 Rail+Sidebar+Main 三列，后来加 FilePanel（文件浏览器），再加 FilePreviewPanel（文件编辑/预览）。每加一个面板就独立写一套拖拽逻辑，互相不感知，导致拖拽体验差、宽度协调有问题。

## 当前问题

### 1. 拖拽互不感知

FilePanel 和 FilePreviewPanel 各自独立管理自己的宽度 ref，拖拽一个时另一个完全不知情，两个面板可以联手把主聊天区挤到 0px。

### 2. 主内容区无最小宽度保护

`sb-main` 只有 `flex: 1`，没有 `min-width`。当右侧面板拖得太宽时，聊天区被压缩到不可用。

### 3. 默认宽度不一致

| 列 | 默认宽度 | 最小 | 最大 |
|---|---|---|---|
| FilePreviewPanel | `window.innerWidth * 0.35`（≥350px） | 280px | 700px |
| FilePanel | 280px | 200px | 600px |

一个用视口百分比，一个用固定 px。窗口缩放时行为不一致。

### 4. 拖拽监听器泄漏

两个组件的 `mousemove` / `mouseup` 都注册在 `document` 上，但 `onUnmounted` 没有清理。

### 5. CSS 分散且不一致

- FilePanel 样式在全局 `main.css`，FilePreviewPanel 样式在 scoped
- 拖拽手柄 active 态一个用 `var(--accent-dim)`，一个用 `var(--accent) !important`
- 类名不统一（`file-panel-drag-handle` vs `panel-drag-handle`）

## 目标布局

```
┌────────┬──────────────┬─────────────────────┬──────────────┬────────────┐
│  Rail  │   Sidebar    │    Main (min 300)   │   Preview    │   Files    │
│  40px  │  0 / 260px   │      flex: 1        │  0 / 280~700 │ 0 / 200~600│
│ fixed  │   toggle     │     auto-grow        │  toggle+drag │ toggle+drag│
└────────┴──────────────┴─────────────────────┴──────────────┴────────────┘
```

- **Rail**：固定 40px，始终可见
- **Sidebar**：toggle 切换 0 ↔ 260px，CSS transition 动画
- **Main**：`flex: 1` + `min-width: 300px`，吸收剩余空间。`min-width` 为 CSS 兜底，真正的约束在 JS 拖拽逻辑中
- **Preview**：由 `panelFile` 控制显隐，宽度可拖拽（280~700px）
- **Files**：由 `collapsed` 控制显隐，宽度可拖拽（200~600px）

Main 的 min-width 会在 JS 层反向传导：拖拽时计算 `剩余空间 = 容器宽 - Rail - Sidebar - 300 - 另一右侧面板宽`，clamp 当前列 ≤ 剩余空间。`ResizeObserver` 监听容器宽度变化，超出时自动收缩右侧面板。

## 实施方案

### Step 1：新建 `usePanelLayout` composable

**文件**：`src/composables/usePanelLayout.ts`

只管理两个可拖拽列的宽度，加共享拖拽函数和容器宽度监听：

```ts
export function usePanelLayout(containerRef: Ref<HTMLElement | null>) {
  const previewWidth = ref(350)
  const filesWidth = ref(280)
  const MAIN_MIN = 300
  const SIDEBAR_WIDTH = 260
  const RAIL_WIDTH = 40

  // ResizeObserver：容器缩小时自动 clamp 右侧面板
  // → 若 previewWidth + filesWidth > 容器宽 - RAIL - SIDEBAR - MAIN_MIN，等比收缩

  // startResize：监听 document mousemove/mouseup，clamp 拖拽列
  // → onUnmounted 清理监听器

  return { previewWidth, filesWidth, startResize }
}
```

**关键设计决策**：

- 手柄留在面板组件内部（保持 DOM 自包含），但拖拽逻辑全走 composable
- `provide` / `inject` 传递宽度和 `startResize`，避免 props 层层透传
- ResizeObserver 防窗口缩放溢出，不是每次拖拽都注册新的

### Step 2：改造 AppShell

- 用 `usePanelLayout` 替代分散的 `panelWidth` ref
- `provide` 宽度给子面板
- `sb-main` 加 `min-width: 300px`（CSS 兜底）

### Step 3：精简两个面板 + CSS 收拢

**FilePreviewPanel**：
- 删除 `panelWidth` ref 和 `onPanelDragStart`
- 通过 `inject` 获取宽度和 `startResize`
- 手柄的 `@mousedown` 绑到注入的 `startResize`
- 手柄 active 态统一为 `var(--accent-dim)`

**FilePanel**：
- 同上，删除内部拖拽逻辑
- 全局样式从 `main.css` 迁入 scoped
- 删 `!important`

### 不变的部分

- 侧栏 Rail/Sidebar 的展开收起逻辑不变
- FilePanel 内部的**垂直分隔条**（文件树 ↔ 内联预览）不动——那是面板内部布局，不是列布局
- Git diff 面板的渲染逻辑不变（仍与 FilePreviewPanel 互斥，走 `v-if/v-else-if`）

## 影响范围

| 文件 | 改动类型 |
|---|---|
| `src/composables/usePanelLayout.ts` | **新建** |
| `src/components/layout/AppShell.vue` | 接入 composable，provide |
| `src/components/shared/FilePreviewPanel.vue` | 删除拖拽逻辑，inject 宽度 |
| `src/components/files/FilePanel.vue` | 删除拖拽逻辑，inject 宽度，CSS 收拢 |
| `src/assets/main.css` | 删除 FilePanel 全局样式 |

## 验收标准

- [ ] 拖拽 Preview 面板时，Files 面板不受影响，Main 不低于 300px
- [ ] 拖拽 Files 面板时，Preview 面板不受影响，Main 不低于 300px
- [ ] 同时打开 Preview 和 Files 时，左侧面板宽度 + Main 300px 有保障
- [ ] 窗口缩放时列宽不溢出（ResizeObserver 自动 clamp）
- [ ] 面板关闭/打开时相邻列平滑过渡
- [ ] vitest 全部通过
- [ ] 无拖拽监听器泄漏（组件销毁后拖动不报错）
